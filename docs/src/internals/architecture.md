# Architecture

## No MATLAB at build time

Mexicah generates MEX files without a MATLAB installation. The key insight:
MEX files are ordinary shared libraries that export a function named
`mexFunction`. The MATLAB C API functions (`mxGetPr`, `mxCreateDoubleMatrix`,
etc.) from `libmx.so` are already present in MATLAB's process when it loads a
MEX file. Mexicah declares these as plain `ccall` references — the dynamic
linker resolves them at MEX load time.

- **Linux**: ELF lazy binding allows undefined symbols in shared libraries.
- **macOS**: `juliac` is invoked with `-undefined dynamic_lookup`.
- **No headers required**: function signatures are declared directly in Julia
  via `ccall`, bypassing the need for `mex.h` or `matrix.h`.

## Build pipeline

```
@mexfunction f(x::T)::R   →   type signature stored in _MEX_EXPORTS

build_mex(f; ...)
  │
  ├─ 1. Validate: hasmethod(f, Tuple{T})
  │                Base.return_types warns if result is Any
  │
  ├─ 2. Codegen: generate_mex_source(...)
  │               → f_mexgen.jl  (complete, self-contained Julia file)
  │
  ├─ 3. Format:  runic -i f_mexgen.jl
  │
  ├─ 4. Compile: juliac --output-lib f_tmp.so --trim=safe --bundle ./mex/ f_mexgen.jl
  │
  ├─ 5. Rename:  mv f_tmp.so → ./mex/f.mexa64
  │
  └─ 6. Setup:   write ./mex/mexicah_setup.m
```

## Generated source structure

`generate_mex_source` produces a file like:

```julia
# AUTO-GENERATED by Mexicah.jl — do not edit by hand.
using Mexicah
import MyModule

Base.@ccallable function mexFunction(
    nlhs::Cint, plhs::Ptr{Ptr{Mexicah.MxArray}},
    nrhs::Cint, prhs::Ptr{Ptr{Mexicah.MxArray}},
)::Cvoid
    Mexicah._mexicah_init_once()
    # argument count checks ...
    _arg1 = Mexicah.load_arg(prhs, 1, Float64)
    _arg2 = Mexicah.load_arg(prhs, 2, Vector{Float64})
    _ret1 = MyModule.f(_arg1, _arg2)
    Mexicah.store_result(plhs, 1, _ret1::Vector{Float64})
    return
end
```

`@ccallable` instructs `juliac` to export `mexFunction` as a C-visible symbol.
With `--trim=safe`, all unreachable Julia code is stripped — the resulting
shared library is 1–3 MB instead of 200+ MB.

## juliac --trim=safe constraints

To pass trim, the generated code (and the user's function) must have:
- No abstract dispatch (all method calls resolve at compile time)
- No `eval` or `@generated` functions in hot paths
- Explicit concrete return type annotations on all contract methods

`@mexfunction` warns if the function's inferred return type is `Any`.

## Runtime sharing

`juliac --bundle` copies `libjulia.so` by its exact soname. When MATLAB loads
multiple MEX files that reference the same soname, the OS dynamic linker maps
the library once and shares it across all MEX files in the process — one GC,
one thread pool, one runtime.
